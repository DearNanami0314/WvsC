- MVP 生命系统：A 版本（受伤只弃牌，先不做销毁/精神伤害）。

- 用户进入实操阶段：需要 MVP 必要内容的实现顺序建议。

## Research Findings
- 入口场景：`project.godot` 指向 `res://scenes/battle/battle.tscn`。
- 全局单例（autoload）：`Events`（事件总线）、`Shaker`（抖动）、`MusicPlayer`、`SFXPlayer`。
- 战斗编排核心：`scenes/battle/battle.gd` 负责回合切换与胜负触发。
- 玩家回合核心：`scenes/player/player_handler.gd` 负责抽牌/弃牌/洗回/回合起止。
- 敌人回合核心：`scenes/enemy/enemy_handler.gd` 串行驱动敌人行动并结束敌方回合。
- 敌人 AI：`enemy_action_picker.gd`（条件动作优先 + 权重随机动作）。
- 卡牌数据模型：`custom_resources/card.gd`（目标类型、费用、效果入口）。
- 牌堆模型：`custom_resources/card_pile.gd`（抽牌、加牌、洗牌、广播数量变化）。
- UI 主协调：`scenes/ui/battle_ui.gd`（手牌与法力 UI，结束回合按钮）。
- 手牌与卡牌交互：`hand.gd` + `card_ui.gd` + `card_state_machine`（拖拽/瞄准/出牌状态流）。
- PDF（根目录 `总结.pdf`）关键信息：
  - 类型定位：卡牌对战 + Roguelike 构筑。
  - 与 STS 相同点：回合制战斗、牌库构筑、随机成长、战斗与事件交替。
  - 关键差异：
	- 以“扑克牌牌型”触发技能（对子/三条/葫芦/顺子/同花等）。
	- 单张可在自由阶段打出，非单张牌型通常每回合一次。
	- J/Q/K 与双王有持有型/被动型/主动型/奥义等差异机制。
	- “牌堆即生命”：就绪牌堆/弃牌堆/销毁牌直接映射生存与上限。
	- 计划存在备牌堆系统（战斗外替换）与无传统背包。
  - 流程偏好：玩家回合拆分为开始→摸牌→自由出牌1→牌型阶段→自由出牌2→弃牌→结束。
  - 长线内容：冒险模式优先，战斗与非战斗约 7:3，地图偏逻辑节点而非可视大地图。
- 风险点：文档中部分规则仍是“暂定”（如手牌上限、Boss打断条件、特殊牌型全集）。
- 教程风格对齐要点（已验证）：
  - 全局事件使用 `Events` Autoload，信号名为过去式/状态式（如 `card_played`, `enemy_turn_ended`）。
  - 编排层采用 `battle.gd` + `player_handler.gd` + `enemy_handler.gd` 三段式。
  - 数据模型以 `custom_resources/*.gd` 的 `Resource` 为核心，运行时通过 `create_instance()/duplicate(true)` 隔离状态。
  - AI 采用 `EnemyActionPicker`（条件动作优先 + 权重动作兜底）。
  - 文件命名与函数命名统一 snake_case，类型命名 PascalCase（`class_name`）。
- 命名规范补充（基于全量扫描）：
  - 文件名：snake_case，与类名对应（`card_ui.gd` ↔ `CardUI`）。
  - `signal`：snake_case，事件完成态命名（`*_started`, `*_ended`, `*_played`）。
  - 导出变量：`@export var name: Type` 强类型优先，常配合 setter（`set = _set_xxx`）。
  - 处理函数：信号回调统一 `_on_xxx`；私有setter `_set_xxx`；常量 `SCREAMING_SNAKE_CASE`。
